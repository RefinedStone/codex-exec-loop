# Changes From `v1.2.9` To Current `prerelease`

This note summarizes the user-visible delta from `v1.2.9` (`b0413ee`) to the current `origin/prerelease` tip (`8b70b5a`).
It is intentionally curated rather than exhaustive.

## Snapshot

- Range: `v1.2.9..prerelease`
- First new commit in the range: `b6f7961`
- Current `origin/prerelease` tip: `8b70b5a`

## Headline Changes

### 1. Shared planning lifecycle commands shipped

Akra now exposes the planning workspace lifecycle both outside and inside the shell.

- External commands:
  - `akra doctor`
  - `akra init`
  - `akra reset <queue|directions|all>`
- In-shell parity:
  - `:doctor`
  - `:init`
  - `:reset`

The external CLI and in-shell flows now share one validation, bootstrap, and reset model.

### 2. First-run planning was simplified

The first accepted planning workspace is no longer framed as a detail-heavy setup ritual.

- `:planning` and `:init` prefer the simple happy path when no planning workspace exists.
- Simple review explains that planning starts without an active next task.
- Queue-idle review and next-task behavior are surfaced during onboarding instead of being implicit.
- Detail authoring remains available, but it is no longer the default first step.

### 3. Resume now restores work context, not just transcript context

Resumed sessions immediately surface planning and queue state.

- Resume status text includes planning status and compact queue framing.
- Shared status projection helpers now feed resume text, footer lines, and queue framing.
- Closed review follow-ups tightened queue-summary edge cases around missing explicit `next_task` values and partial queue summaries.

### 4. Validation expectations became explicit

Terminal validation now uses named profiles instead of loose ad hoc checklists.

- `terminal-baseline`
- `phase1-operator-surface`

Both Bash and PowerShell capture scripts record the selected check profile.

### 5. Supersession first-pass runtime loop shipped

Parallel mode is no longer just a planning concept document. Current `prerelease` already ships the first operator-facing supersession loop.

- `:parallel on` gates entry on git, worktree, `akra`, push, `gh`, and planning readiness.
- The supersession board shows live capability, pool, roster, selected-detail, and distributor projections.
- Queue-driven work can lease one of three `akra` worktree slots, run a main-grade agent session there, and hold completion until hidden planning refresh makes it official.
- Distributor delivery already serializes rebase, push, PR automation, merge integration, and slot cleanup.

### 6. Current branch follow-through beyond `origin/prerelease`

The current branch also implements the repo-scoped authority locator, store-backed drafts and promote, store-backed active planning mutation and queue claims, runtime projection recovery, store-primary read cutover, and legacy bootstrap cleanup.

- Keep that status in `docs/design/` and planning directions until those commits merge back into `prerelease`.
- The release baseline above remains the source of truth for what `origin/prerelease` already ships today.

## Operator-Facing Surface Changes

| Area | `v1.2.9` baseline | Current `prerelease` |
| --- | --- | --- |
| Planning lifecycle | planning lived mostly inside `:planning` flows | shared `akra` and `:` lifecycle commands are shipped |
| Onboarding | planning setup leaned harder on detail-mode framing | simple review is the default first-run path |
| Queue explanation | next-task and queue-idle behavior were less explicit during setup | onboarding explains that accepted queue state drives execution |
| Session continuity | resume restored thread state but not a full planning summary surface | resume immediately shows planning status and queue summary |
| Validation docs | terminal validation existed but without named operator-surface profiles | validation is organized around reusable check profiles |
| Parallel mode | supersession existed mostly as a target architecture lane | supersession now has a shipped readiness gate, control tower, worktree pool, and serial distributor loop |

## Shipped Code Entry Points Added Or Elevated

- Planning reset service: `src/application/service/planning_reset_service.rs`

- Supersession shell entrypoint: `src/adapter/inbound/tui/app/parallel_mode.rs`
