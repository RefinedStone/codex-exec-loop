Network Basics

IP, Port, DNS만 먼저 맞춘다

Example

`https://example.com:443`

이름은 `example.com`, 주소는 DNS가 찾고, 포트 `443`은 TLS 서비스 출입구다.

IP

통신 대상 주소

Port

한 IP 안의 서비스 출입구

DNS

이름을 IP로 바꾸는 시스템

CIDR

대역 표기를 읽을 줄 알면 충분하다

`10.96.0.0/12`

Service CIDR 예시로 자주 만난다.

`192.168.0.0/24`

노드나 로컬 네트워크 예시로 자주 본다.

시험 기준

복잡한 계산보다 "같은 대역인가, 다른 대역인가" 정도만 읽으면 된다.

CKA 네트워크에서는 subnetting 수학보다 리소스 관계와 통신 경로가 훨씬 중요하다.

Transport

시험에서는 대부분 TCP 기준으로 이해해도 된다

항목 TCP UDP
주로 보는 장면 HTTP, Service 연결, `curl` 이번 강의에서는 비중 낮음
이번 수업 기준 핵심 기준 용어 수준만

Three Rules

쿠버네티스 네트워크는 세 문장으로 시작한다

01 모든 Pod는 자기 IP를 가진다
02 Pod 간 통신은 노드가 달라도 가능해야 한다
03 고정 진입점은 Pod가 아니라 Service가 맡는다

Pod View

Pod는 IP를 가지지만 오래 믿을 주소는 아니다

$ kubectl get pods -o wide
NAME         READY   STATUS    IP            NODE
web-7k2m     1/1     Running   10.244.1.12   worker-a
web-9l8z     1/1     Running   10.244.2.07   worker-b

관찰 포인트

Pod IP와 Node를 같이 보면 "같은 노드냐, 다른 노드냐"를 즉시 읽을 수 있다.

시험 포인트

직접 IP 접속과 Service 경유 접속을 구분해야 장애 위치를 줄일 수 있다.

Whole Map

Pod, Service, DNS를 한 장으로 묶어 본다

CoreDNS
ClusterIP Service
node-a pod-a pod-b
node-b pod-c
client pod

사용자는 Service 이름으로 붙고, DNS가 이름을 풀고, Service가 실제 Pod 집합으로 연결한다.

Service

왜 Pod IP 대신 Service를 쓰는가

Without Service

직접 Pod IP 접근

  • Pod 재시작 시 주소가 바뀔 수 있다
  • 복수 Pod 앞단 로드밸런싱이 없다
  • 사용자가 대상 Pod를 직접 알아야 한다
With Service

고정 진입점 + 선택기

  • Service IP와 DNS 이름이 고정 진입점이 된다
  • selector로 Pod 집합을 고른다
  • 대상 Pod가 바뀌어도 사용자는 이름만 쓴다

Service Types

네 가지는 이 차이만 기억하면 된다

ClusterIP

클러스터 내부 기본 타입

NodePort

노드 포트를 통해 외부 진입

LoadBalancer

클라우드 외부 LB 연결

Headless

가상 IP 없이 Pod 레코드를 직접 노출

Selector & Endpoints

Service는 selector로 고르고 endpoints로 드러난다

Service
selector: app=web
Pod
label이 맞으면 선택
Endpoints
선택 결과가 여기 보인다

Service가 안 되면 제일 먼저 `endpoints`를 본다. 비어 있으면 대개 selector 또는 Pod 상태 문제다.

Demo

selector가 틀리면 무엇이 보이는가

$ kubectl get svc web
$ kubectl get endpoints web
$ kubectl describe svc web
$ kubectl get pods -l app=web --show-labels

학생이 봐야 할 것

Service는 있는데 endpoints가 비어 있을 수 있다.

결론

이 장면이면 가장 먼저 selector와 label을 맞춘다.

Namespace

Ingress

Ingress는 밖에서 들어오는 규칙이다

external user
host + path rule
service
pod

Ingress가 직접 Pod를 보지 않는다는 점만 확실히 남기면 시험 문제 해석이 쉬워진다.

Ingress YAML

YAML은 네 줄만 먼저 보면 된다

spec:
  rules:
  - host: shop.example.com
    http:
      paths:
      - path: /api
        backend:
          service:
            name: shop-api
            port:
              number: 80

1. host

어떤 도메인 요청인가

2. path

어느 경로를 매칭하는가

3. service name

뒤에 어느 Service로 보낼 것인가

4. service port

Service의 어느 포트를 쓸 것인가

NetworkPolicy

정책은 누구를 막느냐보다 누구에게 붙느냐부터 본다

`podSelector`

정책이 적용되는 대상 Pod

`policyTypes`

ingress 인지 egress 인지

`from` / `to`

허용하는 출발지와 목적지

NetworkPolicy는 클러스터 전체 방화벽이 아니다. 선택된 Pod에 허용 규칙을 붙이는 리소스다.

Policy YAML

이 블록만 읽어도 절반은 푼다

spec:
  podSelector:
    matchLabels:
      app: backend
  policyTypes:
  - Ingress
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: frontend

대상

`app=backend`인 Pod가 정책 대상이다.

방향

Ingress 규칙만 읽으면 된다.

허용

`app=frontend`에서 들어오는 요청만 허용한다.

Demo

backend는 왜 되고 client는 왜 안 되는가

frontend
backend
client
backend

정책 전

대개 양쪽 다 통신 가능

정책 후

정책이 붙은 backend 입장에서 허용된 출발지만 통과

Drill 1

Pod to Pod가 안 되면 어디부터 볼 것인가

`curl 10.244.2.07:8080`이 실패한다. 첫 확인 순서를 말해 보라.
1Pod Running 여부
2Pod IP와 Node 확인
3다른 Pod에서 직접 IP 접속
4앱 포트와 readiness 확인

Drill 2

Service 문제와 DNS 문제를 분리한다

상황 A. `curl http://web` 실패

첫 명령 두 개

  • `kubectl get svc web`
  • `kubectl get endpoints web`

상황 B. 이름 자체가 안 풀리는 것 같다

첫 명령 두 개

  • `kubectl exec -it client -- nslookup web`
  • `kubectl get pods -n kube-system`

Drill 3

Policy 문제는 이 네 단계로 줄인다

1. 정책 대상 Pod

정책이 누구에게 붙었는가

2. 방향

Ingress 인가, Egress 인가

3. label 매칭

`podSelector`가 실제 label과 맞는가

4. namespace 매칭

`namespaceSelector` 또는 namespace 차이를 봤는가

Final Checklist

시험 직전 이 다섯 문장만 남기면 된다

1. Pod는 변한다

고정 진입점은 Service가 맡는다.

2. Service 문제는 endpoints부터

selector, endpoints, targetPort 순으로 줄인다.

3. DNS 문제는 이름이 IP로 바뀌는가부터

조회가 되면 DNS는 일단 통과다.

4. Ingress도 결국 뒤의 Service와 Pod를 본다

앞단만 보고 끝내지 않는다.

5. NetworkPolicy는 선택된 Pod에 허용 규칙을 붙이는 방식으로 읽는다

누구에게 붙었는지, 어느 방향인지 먼저 본다.

Command Pack
  • `kubectl get pods -o wide`
  • `kubectl get svc` / `get endpoints`
  • `kubectl describe svc` / `describe networkpolicy`
  • `kubectl exec -- nslookup` / `curl`
